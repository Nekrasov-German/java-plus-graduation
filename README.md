# архитектура микросервисов «Explore With Me»
Обзор
Проект «Explore With Me» реализует API‑сервер для работы с событиями и подборками событий. Архитектура построена на микросервисном подходе с использованием API Gateway для маршрутизации запросов.

## Микросервисы
### В системе выделены следующие микросервисы:

1. user-service
   Назначение: управление пользователями.
   Обрабатываемые эндпоинты:
```json
"/admin/users/**"
```
операции администрирования пользователей (получение информации, добавление, удаление).
2. comment-service
   Назначение: работа с комментариями.
   Обрабатываемые эндпоинты:
```json
"/admin/comments/**"
```
администрирование комментариев;
```json
"/users/{userId}/events/{eventId}/comments/**"
```
комментарии к событиям пользователя;
```json
"/events/{eventId}/comments/**"
```
комментарии к публичным событиям.
3. request-service
   Назначение: обработка запросов на участие в событиях.
   Обрабатываемые эндпоинты:
```json
"/users/{userId}/requests/**"
```
заявки пользователя на участие в чужих событиях;
```json
"/users/{userId}/events/{eventId}/requests/**"
```
запросы на участие в событии текущего пользователя (получение и изменение статуса заявок).
4. event-service
   Назначение: центральный сервис для работы с событиями, категориями и подборками.
   Обрабатываемые эндпоинты:

Категории (Admin):
```json
"/admin/categories/**"
```
(добавление, удаление, изменение);
Подборки событий (Admin):
```json
"/admin/compilations/**"
```
(добавление, удаление, обновление);
События (Admin):
```json
"/admin/events/**"
```
(поиск и редактирование событий и их статуса);
События пользователя (Private):
```json
"/users/{userId}/events/**"
```
(получение, добавление, изменение событий пользователя);
Публичные категории:
```json
"/categories/**"
```
(получение категорий и информации о конкретной категории);
Публичные события:
```json
"/events/**"
```
(получение событий с фильтрацией и подробной информации о событии);
Публичные подборки:
```json
"/compilations/**"
```
(получение подборок событий и подборки по ID).
# Маршрутизация через API Gateway
API Gateway перенаправляет запросы к соответствующим микросервисам на основе пути (Path). Ниже — таблица маршрутов:

ID маршрута	URI сервиса	Обрабатываемые пути	Назначение
user_service_route	lb://user-service	/admin/users/**	Управление пользователями (админ)
comment_service_route	lb://comment-service	/admin/comments/**, /users/{userId}/events/{eventId}/comments/**, /events/{eventId}/comments/**	Работа с комментариями
request_service_route	lb://request-service	/users/{userId}/requests/**, /users/{userId}/events/{eventId}/requests**	Запросы на участие в событиях
event_service_route	lb://event-service	/admin/categories/**, /admin/compilations/**, /admin/events/**, /users/{userId}/events/**	Админ‑операции и приватные события
categories_event_service_route	lb://event-service	/categories/**	Публичные категории
events_event_service_route	lb://event-service	/events/**	Публичные события
compilations_event_service_route	lb://event-service	/compilations/**	Публичные подборки событий
Ключевые особенности архитектуры
Централизованная маршрутизация. Все запросы проходят через API Gateway, который определяет целевой микросервис по пути запроса.

Балансировка нагрузки. Префикс lb:// в uri указывает на использование балансировщика нагрузки (load balancer) для распределения запросов между экземплярами сервиса.

Разделение зон ответственности:

user-service отвечает только за пользователей;

comment-service — только за комментарии;

request-service — за запросы на участие;

event-service объединяет функциональность, тесно связанную с событиями, категориями и подборками, что снижает межсервисные вызовы.

Гибкость масштабирования. Микросервисы можно масштабировать независимо. Например, event-service, обрабатывающий публичные эндпоинты, может требовать большего числа экземпляров из‑за высокой нагрузки.

Чёткое разделение API:

Public — общедоступные данные (события, категории, подборки);

Private — операции для авторизованных пользователей (управление своими событиями, заявками);

Admin — функции администрирования.

# Начало работы
## Запустите API Gateway и все микросервисы.

Убедитесь, что сервисы зарегистрированы в Service Discovery (например, Eureka), если используется динамическая маршрутизация.

Отправьте запрос к API Gateway по адресу http://localhost:8080, указав нужный эндпоинт. Gateway автоматически перенаправит запрос в соответствующий сервис.
